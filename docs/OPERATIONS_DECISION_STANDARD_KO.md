# 운영 의사결정 표준 (Runbook)

## 1. 목적/범위
이 문서는 `order-inventory-api-tuning`의 실험 결과를 운영 의사결정에 바로 적용하기 위한 표준입니다.
대상은 주문 생성/조회 경로와 DB·ORM·동시성 튜닝 항목(인덱스, 파티셔닝, ID 전략, N+1, 풀/스레드)입니다.

---

## 2. 핵심 SLO/SLI 정의
- **핵심 SLO(기본)**
  - `http_req_duration p95` < 300ms (일반 조회)
  - `http_req_duration p95` < 1500ms (고부하 주문/마이페이지)
  - `http_req_failed` < 0.1%
- **필수 관찰 지표**
  - k6: p95/p99, req/s, failed
  - 애플리케이션: CPU p95, GC pause
  - DB: CPU p95, 활성 커넥션
  - Hikari: active/idle/**pending**

> 참고: 현재 `orders/me` 실험에서는 p95가 3~4.7초 수준으로 높고, Hikari pending이 68~179까지 관측되어 병목이 조회 경로+풀 대기에서 동시에 발생하는 패턴입니다.

---

## 3. 의사결정 트리 (요약)

### Step A. 실패율이 먼저 올라가는가?
- **YES** → 인덱스/락/풀 포화 우선 점검
- **NO** → 다음 단계로

### Step B. p95/p99만 치솟고 평균은 낮은가? (tail latency 문제)
- **YES** → Fetch Join 남용/row 폭증, 풀 pending, GC burst 의심
- **NO** → 전체 처리량 병목(CPU/DB) 점검

### Step C. 쓰기 경로가 병목인가, 읽기 경로가 병목인가?
- **쓰기 병목**: 파티셔닝, ID 전략, 배치 전략 우선
- **읽기 병목**: EntityGraph+BatchSize, 페이지네이션 전략, 인덱스 우선

---

## 4. 시나리오별 표준 적용안

## 4.1 주문 생성(Write) 병목
### 증상
- p95/p99 상승 + 쓰기 부하 구간에서 dropped iteration/큐 대기 증가

### 적용 우선순위
1) **파티셔닝 적용 검토 (시간축 대량 write인 경우)**
- 근거: `np` 평균 12.19ms → `pt` 평균 10.21ms(약 16% 개선).
- 기준: 월/주 단위로 데이터가 빠르게 누적되고, 최신 구간 write hotspot이 뚜렷할 때 적용.

2) **ID 전략 재선정 (IDENTITY → SEQUENCE 후보)**
- 근거: IDENTITY 대비 SEQUENCE에서 p95/p99가 유의미하게 낮음.
- 단, SEQUENCE 실험에서 Hikari pending(176) 관측됨 → 풀/스레드 동반 튜닝 필수.

3) **풀/스레드 동시 튜닝**
- 규칙: ID 전략 바꿀 때는 반드시 pool/thread를 한 세트로 재측정.

### 롤백 조건
- 변경 후 p99가 20% 이상 악화되거나 pending 급증 시 즉시 이전 조합으로 롤백.

---

## 4.2 주문 상세/마이페이지(Read) 병목
### 증상
- 평균은 괜찮은데 p95/p99가 비정상적으로 증가
- Hikari pending 증가, DB CPU 상한 접근

### 적용 우선순위
1) **EntityGraph + BatchSize를 기본 전략으로 채택**
- 근거: `order-detail` 비교에서 fetch 전략이 light/heavy 모두 p95/p99 악화 및 오류 증가 경향.
- 표준: 1:N 다건 연관 조회는 Fetch Join을 기본값으로 두지 않음.

2) **Fetch Join 사용 조건 제한**
- 조건: 결과 cardinality가 작은 구간(작은 fan-out) + 페이지 부작용이 없는 경우에만 제한적 사용.

3) **페이지네이션 전략 보강**
- `orders/me`처럼 p95가 수초 단위이면 offset 기반 페이지네이션/과도한 조인을 의심.
- 개선: keyset pagination, projection 전용 조회, read model 분리 검토.

### 롤백 조건
- fetch 전략 전환 후 p95 악화 또는 errors 증가 시 즉시 이전 전략 복귀.

---

## 4.3 커넥션 풀/스레드 튜닝
### 증상
- 처리량은 오르는데 tail latency와 오류율이 함께 악화

### 적용 우선순위
1) **풀 크기 무작정 증가 금지**
- 근거: light 테스트에서 pool 30은 p99가 pool 10/20보다 악화.

2) **스레드 증설은 50→100→200 단계 실험 후 선택**
- 근거: heavy 테스트에서 스레드 증설로 req/s는 증가했지만 구간별 수익 체감 존재.

3) **최적화 목표는 최대 RPS가 아니라 SLO 충족율**
- 표준 점수 예시: `0.5*p95 + 0.3*error + 0.2*pending` 최소 조합 선택.

---

## 5. 표준 임계치 (권고)
- **Hikari pending**
  - 정상: < 20
  - 주의: 20~100
  - 위험: > 100 (즉시 풀/쿼리/스레드 재조정)
- **p99/p95 비율**
  - 정상: < 1.8
  - 위험: >= 2.0 (tail 불안정)
- **오류율**
  - 위험: >= 0.1%

---

## 6. 운영 변경 절차 (표준 프로세스)
1. **가설 작성**: 병목 가설 1개만 선택 (예: N+1로 tail 증가)
2. **변수 1개 변경**: 인덱스/그래프/풀 중 1개만 변경
3. **동일 부하 재실행**: 동일 RPS, 동일 데이터 상태
4. **비교 판정**: p95/p99/error/pending 4지표 비교
5. **승인/롤백**: 임계치 충족 시 채택, 미충족 시 롤백
6. **문서화**: 결과를 md로 남기고 커밋 SHA/설정값 기록

---

## 7. 이번 저장소 기준 권장 기본값
- **읽기 기본 전략**: EntityGraph + batch fetch
- **쓰기 기본 전략**: 파티셔닝 고려 + ID 전략은 트래픽 특성별 선택
- **동시성 기본값**: pool=20 부근에서 시작, thread=100 기준으로 부하/대기 확인
- **인덱스 정책**: 조건절+정렬 복합 인덱스 우선

---

## 8. 부록: 근거 데이터 요약
- 인덱스 적용 시 RPS/실패율 대폭 개선
- 파티셔닝 시 평균 응답 개선(12.19 → 10.21ms)
- SEQUENCE가 IDENTITY 대비 p95/p99 개선, 단 pending 리스크 존재
- order-detail에서 fetch 전략이 tail/오류 측면 불리
- orders/me 구간은 전반적으로 p95가 수초 단위로 고비용 경로
